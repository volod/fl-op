## Problem Statement

An agricultural enterprise providing field services to farmers needs a decision
support system for complex multi-resource scheduling and dispatch. The operator
manages a heterogeneous fleet (3000-5000 vehicles, 20000 implements, 50 depots)
and must optimally assign vehicle-implement pairs to farmer orders, respecting:

- Vehicle-implement compatibility (hitch type, PTO, hydraulics, ISOBUS, power)
- Weather windows (soil moisture, rain, wind, spraying restrictions)
- Contract terms (deadlines, penalties, SLAs, farmer priority)
- Inventory constraints (fuel, fertilizers, chemicals per depot; each vehicle
  carries sufficient supply for one job; mobile tanker top-ups modeled as a
  fixed time penalty at depot)
- Geographic routing (vehicle and implement current location to field)
- Operator assignment (one operator per vehicle per shift, hard constraint)
- Rolling-horizon re-optimization (breakdowns, weather changes, cancellations)

The mathematical class is: Heterogeneous Fleet VRP with Time Windows (HFVRPTW)
+ Multi-resource Scheduling + Profit-Maximizing Order Selection.

Objective: Maximize profit = contract revenue - fuel cost - labor cost -
fertilizer/chemical cost - machine operating cost - repositioning cost -
delay penalties - risk buffer.

Some orders may be rejected (not served) if assigning them reduces total
profit or causes higher contractual penalties elsewhere. Unserved orders are
marked as rejected; their contract penalties are added to total cost in KPIs.

## What Makes This Cool

A single agricultural enterprise runs a combinatorial optimization problem at
scale that rivals airline crew scheduling in complexity. The open-source POC
will be the first public implementation of a complete decision support pipeline
for agricultural fleet optimization:

1. Realistic synthetic data generator for the full agricultural domain model
2. Hierarchical OR-Tools solver handling 3000+ vehicle fleets via depot clustering,
   with parallel cluster solving (one process per CPU core)
3. Rolling-horizon reschedule with plan-diff output (old vs. new schedule)
4. CLI-first interface making the solver accessible without a full ERP stack

Most academic VRP implementations ignore inventory, weather windows, and
vehicle-implement compatibility simultaneously. This does all three.

## Constraints

- Python 3.10+, uv for dependency management (pyproject.toml)
- OR-Tools as primary solver (pip-installable, no JVM dependency)
- OR-Tools routing library handles time-window scheduling and route
  sequencing in a single model; no A/B test in the POC
- Pyomo as fallback only for constraints OR-Tools cannot model
- Timefold is explicitly excluded from the POC (JVM stack complexity)
- Output artifacts written to .data/<command-name>/<run_timestamp>/
- Every run logged; old schedules preserved for plan-diff comparison
- CLI entry point via src/fl_op/main.py; shell wrappers in scripts/
- --latest shorthand: all commands that accept a --data or --schedule path
  support --latest as an alias that resolves to the most recently created
  timestamp directory under .data/<command>/ (e.g. fl-op solve --data latest)
- Flag naming: --data (dataset root) used by solve, reschedule, query-contract;
  --data-path (real CSV import) used by generate-data only; --schedule (path to
  schedule.json); all accept --latest shorthand
- File path inputs (--data, --schedule, --event, --contract, --data-path) are
  resolved via pathlib.Path.resolve() and validated to be within the current
  working directory or .data/; raise click.BadParameter with message
  "[error] <what>. <why>. Try: <fix command>." format for all CLI errors
- Risk enum values: consistently "low"/"medium"/"high" across all output fields
  (weather_risk, schedule_conflict_risk, contractual_risk); "med" is forbidden
- JSON artifacts include schema_version: "1.0" and run_metadata:
  {timestamp, command, args, fl_op_version} in every root output file
- schedule_kpis.json written alongside schedule_report.txt with all KPI fields
  as machine-readable JSON (for downstream tooling and dashboards)
- No magic numbers: all constants in src/fl_op/core/constants.py
- Cluster solvers run in parallel via multiprocessing.Pool

## Premises

1. SCOPE: The immediate deliverable is a Python CLI POC with 4 commands:
   generate-data, solve, reschedule, query-contract. Not a full platform.
   The POC validates the optimization approach before any UI or API is built.
   AGREED.

2. SOLVER: OR-Tools routing library is the primary solver (handles time-window
   scheduling and route sequencing in one model). Pyomo is fallback only.
   No multi-solver hybrid in the POC. AGREED.

3. SCALE: The demo must handle production scale from day 1: 3000-5000 vehicles,
   20000 implements, 2500-10000 concurrent orders, 50 depots.
   This requires hierarchical decomposition with parallel solving. CORRECTED
   (from original premise: small synthetic data is not sufficient).

4. OBJECTIVE: Primary objective is profit maximization with hard constraints
   (compatibility, weather, time windows, operator shifts). Risk fields in
   output (weather_risk, contractual_risk) are informational only; they do
   not affect the optimization weights. AGREED.

5. STACK: Python 3.10+ with uv, OR-Tools via pip, multiprocessing for
   parallel cluster solving, NumPy/scipy for data generation. AGREED.

6. WEATHER & BREAKS: Reschedule command runs the solver on remaining unstarted
   orders with updated state; already-started orders are frozen.
   Old schedule is preserved; output includes a plan-diff. AGREED.

## Approaches Considered

### Approach A: Hierarchical Decomposition with OR-Tools (SELECTED)

Pre-filter vehicles/implements by compatibility matrix, cluster orders by
depot/geographic region, run OR-Tools solver on each cluster in parallel
(50-300 orders per cluster, manageable subproblem), aggregate into a
dispatch plan.

- Effort: M (3-6 weeks for POC)
- Risk: Medium (clustering quality affects solution quality; inter-depot
  moves not globally optimal)
- Key strength: scales linearly with depots, not exponentially with fleet size
- Pre-filtering reduces search space from 3000x20000 to ~50-500 per order
- Parallelism: 50 clusters * 60s each = 60s wall time on a 50-core machine;
  ~5-10min on a typical 8-16 core dev machine

### Approach B: Monolithic OR-Tools + LNS

Single global CP-SAT solve with Large Neighborhood Search, time-limited.
Not chosen: memory/time blow-up at 3000+ vehicles; non-deterministic quality.

### Approach C: Two-Stage Greedy + Local Search

Fast greedy assignment + swap-move improvement. Reserved as warm-start
generator for Approach A: greedy solution is passed to OR-Tools as an initial
hint before LNS, improving convergence within the time limit.

## Recommended Approach

**Hierarchical Decomposition with OR-Tools (Approach A)**

Implementation layers in execution order:

```
Layer 0: Data Model + Compatibility Matrix
  Entities: Vehicle, Implement, Order, Field, Depot, Material, Weather,
            Operator, Contract (Pydantic models in src/fl_op/models/)
  Vehicle: ID, type, power_kW, fuel_rate_L_ha, hitch_type, PTO_kW,
           hydraulics, ISOBUS, speed_km_h, location, availability,
           cost_per_hour, maintenance_status, assigned_operator
  Implement: ID, type, compatible_hitch, required_power_kW, working_width_m,
             productivity_ha_h, material_compatibility, calibration_norms
  Operator: ID, shift_start, shift_end, licensed_vehicle_types, assigned_vehicle
            (one operator per vehicle per shift, hard constraint; pool size
            configurable in generator)
  Order: farmer, field_polygon, crop, operation_type (OperationType enum),
         area_ha, deadline, earliest_start, contract_price, penalty_per_day,
         priority, inputs_required
  OperationType: Python Enum (SPRAYING, TILLAGE, SEEDING, HARVESTING, FERTILIZING)
    prevents silent string-mismatch failures in compatibility filter
  Implement.type: Python Enum (same domain values); compatibility filter
    compares enums, not strings
  Compatibility matrix: numpy.ndarray(dtype=bool) shape (N_vehicles, N_implements)
    ~100 MB at 3000x20000; serialized as numpy.memmap .npy file; workers load
    via np.load(mmap_mode='r') for zero-copy access without pickling overhead;
    power_margin_pct stored as separate float32 memmap file of same shape
  Pydantic circular refs (Vehicle<->Operator, Order<->Contract): use
    TYPE_CHECKING imports + model_rebuild() after all models loaded; never
    use `from __future__ import annotations` (banned by CLAUDE.md)
  Layer interface contracts (src/fl_op/models/types.py): TypedDict definitions
    for ClusterSpec, FeasibleAssignment, DispatchPackage, InfeasibleOrder;
    all pipeline functions typed with these; validated at layer boundaries
    (before Pool.submit and after worker return) to prevent silent dict drift
  Weather: forecast by field, rain_mm, wind_ms, soil_moisture_pct,
           temperature_C, spraying_suitability (bool), tillage_suitability (bool)

  Output: .data/generate-data/<ts>/schema/ with Pydantic JSON schemas

Layer 1: Synthetic Data Generator (generate-data command)
  Usage: fl-op generate-data --vehicles N --implements N --orders N
         --depots N [--seed S] [--data-path DIR]
  Defaults: --vehicles 1500 --implements 6000 --orders 2500 --depots 500;
            every default can be overridden by passing the corresponding CLI flag
  --data-path DIR: if provided, load real fleet data from DIR instead of
    generating synthetic data; DIR must contain vehicles.csv, implements.csv,
    orders.csv, depots.csv in the standard schema; missing fields default to
    synthetic fill; allows direct validation against real historical schedules
  Implementation: NumPy vectorized sampling (no row-by-row Python loops);
                  geographic depot/field placement via scipy.spatial.BallTree
                  with haversine metric (correct for large lat/lon regions;
                  Euclidean cKDTree is wrong at agricultural scale);
                  realistic agricultural distributions (power curves,
                  seasonal demand, contract price/penalty)
  Operator pool: default 1.5x vehicle count; each operator assigned to
                 one shift per day
  Target: 60s wall time at --vehicles 1500 --implements 6000 --orders 2500
          --depots 500
  Output: .data/generate-data/<ts>/ with CSV + JSON datasets

Layer 2: Pre-Filter + Clustering + Global Pre-Allocation (internal pipeline step)
  Per order: filter vehicles where power_kW >= implement_required_power *
             terrain_factor; filter implements by OperationType enum suitability
  Depot affinity: scipy.spatial.BallTree with haversine metric; each order
    assigned to nearest depot by great-circle distance (straight-line for POC;
    travel-time refinement is post-POC tuning)
  Cluster orders by depot affinity
  Global pre-allocation pass (src/fl_op/solver/allocation/):
    sort clusters by sum(order.penalty_per_day) descending (highest total
    penalty exposure first — not raw order count; this prevents a 51-order
    cluster of low-value jobs from starving a 50-order cluster with deadline-
    critical high-penalty contracts); tiebreak by cluster_id for determinism
    iterates clusters priority-first; reserves each implement and operator to
    the first cluster that claims it; removes reserved resources from all
    subsequent clusters' feasible sets; prevents cross-cluster double-assignment
    of a physical implement or a licensed operator; runs in O(N_implements)
    before any cluster solver starts
  V-I pair collapsing: after pre-allocation, for each physical vehicle_id within
    a cluster, keep only its highest-scoring V-I pair (by greedy score) as the
    routing model vehicle; prevents one physical tractor appearing in two routes
    within the same cluster with different implements
  Each feasible vehicle-implement pair is treated as a distinct vehicle in
    the OR-Tools routing model for that cluster. After pre-filtering and
    pre-allocation, expected pairs per cluster: 50-500
  Output: feasible_assignments.json per cluster (in-memory, not on disk)

Layer 3: OR-Tools Solver per Cluster (core of solve command)
  Pre-routing V-I cap: cap to top MAX_PAIRS_PER_ORDER (constant, default 30)
    V-I pairs per order ranked by greedy score before routing model construction;
    at 500 V-I pairs x 200 orders OR-Tools callback pressure is prohibitive;
    this reduces routing model vehicle count to <=30*200=6000 max per cluster,
    which the library handles within 60s with greedy warm-start
  Guard: if cluster dict has zero feasible pairs after pre-allocation and
    V-I collapse, return ([], mark_all_as_infeasible(cluster, "no_compatible_vehicle"))
    immediately without constructing a routing model
  Warm-start: run greedy assignment first (NumPy vectorized scorer over all
    feasible V-I pairs for all orders -- 5M+ evaluations must be vectorized,
    not Python loops); pass result to OR-Tools as initial solution hint
  Solver: OR-Tools routing library per cluster; handles time-window scheduling
    and route sequencing in a single model; no separate CP-SAT sub-problem
  Time windows: from contract deadlines + operator shifts (hard constraints)
  Resource-uniqueness: AllowedVehicles or vehicle-incompatibility lists in the
    routing model ensure each V-I pair appears in at most one route per cluster;
    same physical vehicle or implement cannot be assigned to two concurrent jobs
  Inventory: modeled as resource dimensions on routing arcs (fuel + fertilizer
             per arc; depot restocking at fixed time cost; mobile tanker = depot
             variant with location variable, simplified as nearest cluster depot)
  Operator: each vehicle-implement pair must have an available operator for
            the shift; modeled as a vehicle capacity constraint (operator-hours)
  Time limit: configurable per cluster (default: 60s)
  Threading: num_search_workers=1 per worker process; prevents CPU
    over-subscription when Pool runs one process per core
  Process boundary: routing model created and destroyed inside the worker
    function; no Pydantic models or OR-Tools objects cross process boundaries;
    worker accepts and returns plain Python dicts only
  Worker return: tuple (dispatch_packages: list[dict], infeasible_orders: list[dict])
    -- infeasible orders MUST be returned explicitly; never silently dropped
  Parallelism: multiprocessing.Pool(start_method="spawn"); one OR-Tools
    process per CPU core; maxtasksperchild=1 to release memory after each
    cluster; per-task timeout via Pool.apply_async + AsyncResult.get(timeout=);
    on WorkerLostError or TimeoutError: mark all orders in that cluster as
    infeasible with reason "solver_timeout"; ~50 clusters / N_cores = wall time
  Output per cluster: dispatch_package[]
    { vehicle_id, implement_id, operator_id, order_id, route, start_time,
      finish_time, required_fuel_L, required_fertilizer_kg, refueling_points,
      weather_restriction_violated: bool, weather_risk: str ("low"|"med"|"high"),
      contractual_risk: str, gross_margin_EUR, notes }
  Rejected orders: orders infeasible within time limit -> infeasible_orders.json
    { order_id, reason: "violated_constraint_name", detail: str }

Layer 4: Aggregation + Report (output of solve command)
  Merge cluster dispatch packages; compute global KPIs: total_revenue_EUR,
  total_cost_EUR, total_margin_EUR, orders_served_count, orders_rejected_count,
  weather_risk_exposure_pct, average_vehicle_utilization_pct,
  greedy_baseline_margin_EUR (margin from greedy warm-start solution alone,
    for comparison — lets operator see solver improvement over naive assignment)
  Output: .data/solve/<ts>/
    schedule.json (full dispatch packages)
    schedule_report.txt (human-readable summary + KPIs)
    infeasible_orders.json (rejected orders with constraint violations)
  Exit code: 0 if any orders served, 1 if all orders infeasible

Layer 5: Rolling-Horizon Reschedule (reschedule command)
  Usage: fl-op reschedule --schedule .data/solve/<ts>/schedule.json
         --event <event.json>
  Event schema: { type: "weather"|"breakdown"|"cancellation"|"mark_started",
                  affected_vehicle_id? str, order_id? str,
                  new_weather_forecast? WeatherUpdate, timestamp: str }
  "mark_started" event: sets order.status = "started" for the given order_id;
    subsequent reschedule calls will freeze that order
  Freeze orders with status "started" (cannot be changed)
  Re-run Layers 2-4 on remaining orders with updated state (vehicle location,
  weather, inventory, operator availability)
  Guard: if remaining_orders is empty after freezing (all orders started),
    write empty schedule.json + plan_diff.json with "no_unstarted_orders"
    summary and exit 0; do not construct BallTree on empty order list
  Output: .data/reschedule/<ts>/
    schedule.json (new schedule)
    plan_diff.json (structured: changed_assignments[], delayed_orders[],
                    margin_delta_EUR, freed_resources[], summary_narrative)
    plan_diff.txt (human-readable summary generated from plan_diff.json)
  Exit code same as Layer 4

Layer 6: New Contract Query (query-contract command)
  Usage: fl-op query-contract --data .data/generate-data/<ts>/
         --schedule .data/solve/<ts>/schedule.json
         --contract new_contract.json
  new_contract.json required fields:
    { farmer_id, field_polygon, crop, operation_type, area_ha, deadline,
      earliest_start, contract_price_EUR, penalty_per_day_EUR, priority,
      inputs_required: [{ material_type, quantity_kg }] }
  Implementation: runs Layer 2 pre-filter on the new order; greedy top-3
                  selection from feasible pairs ranked by estimated margin;
                  no OR-Tools solver call; compatibility matrix lookup only
  schedule_conflict_risk computation: for each top-3 option, scan schedule.json
    for any dispatch_package using the same vehicle_id or implement_id with a
    time window overlapping [estimated_start, estimated_finish]; if overlap found
    -> "high"; if within 2h -> "medium"; else -> "low"
  Time budget: dominated by matrix lookup, expected <5s at production scale
  Output: 3 ranked options:
    { rank, vehicle_id, implement_id, operator_id, estimated_start,
      estimated_finish, estimated_fuel_L, estimated_fertilizer_kg,
      estimated_margin_EUR, schedule_conflict_risk: "low"|"medium"|"high",
      recommendation_note }
```

## Infeasibility Output Contract

When any order cannot be assigned:
- `infeasible_orders.json` is always written (empty array if none)
- Each entry: `{ order_id, reason: "<constraint_name>", detail: "<human text>" }`
- Constraint names: "no_compatible_vehicle", "weather_restriction",
  "time_window_breach", "insufficient_inventory", "no_available_operator",
  "solver_timeout"
- Exit code: 0 (partial success) if at least one order was served; 1 if none

2. `fl-op solve --data .data/generate-data/<ts>/` completes in under 10 minutes

  synthetic scenarios are sufficient to validate the scheduling logic.

  the algorithmic approach.

  depot variant with fixed time penalty; full inventory-routing adds 2-3 weeks and is
  not needed to validate the dispatch pipeline.
- **Operator overtime costing**: shifts are hard constraints in POC (no work past
  shift_end); overtime as a soft cost with penalty is post-POC configuration.
- **UI / REST API layer**: CLI-first is the agreed interface; no web frontend or API
  gateway in the POC.
- **PyPI package / Docker image / CI**: post-POC distribution; not needed to validate
  the solver approach.
- **Multi-solver hybrid (OR-Tools + Timefold)**: explicitly excluded; JVM stack
  complexity outweighs marginal quality gain for a POC.
- **A/B testing between solver configurations**: single configuration per POC run.
- **Cross-depot vehicle repositioning optimization**: vehicles start at their current
  location and travel to the nearest feasible field; global repositioning across depots
  is post-POC.

## Failure Modes

One realistic production failure per new codepath:

- **Layer 1 generator / BallTree**: coordinates passed in degrees but BallTree
  configured for radians (or vice versa) -- silent bad clustering where all orders
  map to the same depot. Mitigation: assert coordinate range (-90..90, -180..180)
  before BallTree construction.

- **Layer 2 pre-filter / OperationType enum**: a legacy CSV row with a raw string
  operation_type that does not match the Enum -- Pydantic raises ValidationError at
  load time (not silently at filter time). Mitigation: test T02 catches this; fail
  fast at data load, not mid-solve.

- **Layer 2 pre-allocation / tie-breaking**: all clusters happen to have equal order
  counts (e.g. perfectly balanced synthetic data) -- sort is non-deterministic and
  the same input produces different plans on re-run. Mitigation: secondary sort key
  is cluster_id (stable tiebreak); document in core/constants.py.

- **Layer 3 cluster solver / worker OOM**: a cluster with 500+ V-I pairs at 60s limit
  causes worker process to be killed by OS OOM killer -- Pool.get() raises
  WorkerLostError with no dispatch packages or infeasible orders for that cluster.
  Mitigation: wrap Pool.get() in try/except WorkerLostError; mark all orders in the
  failed cluster as infeasible with reason "solver_timeout".

- **Layer 3 cluster solver / no feasible solution**: routing library finds no solution
  within time limit -- must return ([], infeasible_orders_list) tuple, not raise.
  Mitigation: check solver status before extracting routes; test T18 covers this path.

- **Layer 4 aggregator / wrong tuple length**: a future code change returns only
  dispatch_packages from a worker (drops infeasible element) -- aggregator silently
  loses all rejected orders. Mitigation: assert len(worker_result) == 2 at aggregator
  entry; test T24 cross-checks infeasible count.

- **Layer 5 reschedule / frozen order detection**: order.status field missing or
  misspelled in input schedule.json -- orders wrongly treated as unstarted and
  re-assigned. Mitigation: Pydantic validator on OrderStatus enum; raise on unknown
  value, do not default to "pending".

- **Layer 6 query / tie in margin estimates**: two V-I pairs have identical estimated
  margin -- top-3 list changes order across runs. Mitigation: secondary sort by
  vehicle_id (stable tiebreak); test T31 asserts deterministic order.

## Worktree Parallelization Strategy

Build steps and their dependencies (from build order, 0-indexed):

```
Step  Module                              Depends on
  1   constants.py                        (none)
  2   models/ (Pydantic + numpy matrix)   1
  3   data/generator.py                   2
  4   solver/preprocessing.py             2, 3
  5   solver/allocation/                  4
  6   solver/greedy.py                    4
  7   solver/cluster_solver.py            4, 5, 6
  8   solver/aggregator.py + solve CLI    7
  9   solver/reschedule.py                8
 10   solver/query.py                     4, 6
 11   tests/ + smoke test                 all
```

Parallel lanes (each lane is a git worktree):

- **Lane A (critical path)**: 1 -> 2 -> 3 -> 4 -> 6 -> 7 -> 8
- **Lane B (pre-allocation)**: branches at step 4; builds step 5; merges into 7
- **Lane C (reschedule + query)**: branches at step 8; builds steps 9 + 10 in parallel
- **Lane D (tests)**: branches at step 4 for unit tests of layers 0-2; expands as
  each module lands; smoke test wired last

Maximum parallelism point: after step 8 lands, lanes C and D can both proceed
independently. On a solo developer, serial order A -> B -> C -> D is equivalent;
the lane split pays off when two engineers work simultaneously.
