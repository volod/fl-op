# Test Plan — fl-op POC
Generated by /plan-eng-review on 2026-05-21
Branch: main
Repo: fl-op

## Framework
pytest; tests/ directory; smoke test command: `pytest tests/ -x -q`

## Affected Commands
- `fl-op generate-data` — synthetic data generation
- `fl-op solve` — hierarchical OR-Tools cluster solve
- `fl-op reschedule` — rolling-horizon re-optimization
- `fl-op query-contract` — new contract feasibility query

## Key Interactions to Verify

### Data Model (Layer 0)
- Pydantic rejects invalid field types at parse time (not at filter time)
- OperationType enum rejects unknown strings immediately
- CompatibilityMatrix is numpy bool (verify dtype, shape, memory usage)
- Vehicle<->Operator circular reference: model_rebuild() works; no ImportError at runtime

### Pre-Filter (Layer 2)
- Only pairs where vehicle.power_kW >= implement.required_power_kW * terrain_factor pass
- Order with zero feasible pairs -> infeasible_orders entry with reason="no_compatible_vehicle"
- Every order appears in exactly one cluster (no duplicate, no missing)

### Cluster Solver (Layer 3)
- Worker function: create OR-Tools RoutingIndexManager inside child process (not passed as arg)
- Worker function: return value is plain Python tuple (list[dict], list[dict]), not OR-Tools objects
- Time windows: dispatch start_time >= order.earliest_start for all dispatched orders
- Fuel: sum(required_fuel_L per route arc) <= vehicle fuel capacity
- Solver exits within configured time_limit_seconds + 5s (no infinite hang)

### Aggregation (Layer 4)
- No duplicate order_id across merged dispatch packages from different clusters
- infeasible_orders.json contains orders rejected by ANY cluster, not just last cluster
- Exit code 1 only when zero orders are dispatched

### Reschedule (Layer 5)
- Orders with status="started" appear unchanged in new schedule (frozen)
- plan_diff.txt lists orders that changed depot assignment
- .data/solve/<ts>/schedule.json is preserved; reschedule writes to .data/reschedule/<ts>/

### Query-Contract (Layer 6)
- No RoutingModel or RoutingIndexManager instantiated during query (grep test)
- Returns exactly 3 options (or all feasible pairs if fewer than 3 exist)
- Results sorted by estimated_margin_EUR descending

## Edge Cases
- All orders infeasible: exit code 1, infeasible_orders.json populated, no crash
- Empty cluster (0 orders assigned to a depot): solver returns immediately with empty dispatch
- Single-vehicle fleet: solve completes, query returns 1 option
- Weather event: all fields restricted -> all orders infeasible on reschedule
- New contract with no feasible vehicle: query returns 0 options with clear error message

## Critical Paths (E2E smoke tests)
1. `fl-op generate-data --vehicles 50 --implements 200 --orders 20 --depots 5 --seed 42`
   - Verify: .data/generate-data/<ts>/vehicles.json exists and has 50 entries
   - Verify: all entries pass Pydantic validation

2. `fl-op solve --data .data/generate-data/<ts>/`
   - Verify: .data/solve/<ts>/schedule.json exists and is valid JSON
   - Verify: .data/solve/<ts>/infeasible_orders.json exists (may be empty)
   - Verify: at least 1 order dispatched (orders_served_count > 0)

3. `fl-op reschedule --schedule .data/solve/<ts>/schedule.json --event weather.json`
   - Verify: .data/reschedule/<ts>/plan_diff.txt exists
   - Verify: original .data/solve/<ts>/schedule.json is unchanged

4. `fl-op query-contract --data .data/generate-data/<ts>/ --schedule .data/solve/<ts>/schedule.json --contract test_contract.json`
   - Verify: output JSON has rank field 1, 2, 3
   - Verify: options sorted by estimated_margin_EUR desc
